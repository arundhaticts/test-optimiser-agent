## Golden rule of sequencing

Build bottom-up (data shape → tools → NLP → nodes → graph → entrypoint → tests), so every
layer can be tested against the layer below before the graph exists. The three "Blocker
Fixes" are built and tested as first-class work in Phases 6 and 9 — not optional polish.

## Phases in detail (how to pick up each one)

### Phase 0 — Foundations & learning the mechanics ✅
Working environment + understanding of LangGraph before touching the agent.
**Do:** create/activate `.venv`; `pip install -r requirements.txt`;
`python -m spacy download en_core_web_sm` (only needed if you set `SPACY_NER=1`); copy
`.env.example`→`.env`, add `GEMINI_API_KEY`; run the three `learning/` scripts in order
(counter → conditional branch → interrupt/resume).
**Exit check:** all three learning scripts run; you can explain how `interrupt()` pauses and
resumes carrying the human's answer.

### Phase 1 — State & configuration ✅
**Files:** `src/state.py`, `src/config.py`, `tests/test_state.py`.
`TestOptimiserState` matches the spec's Typed State Schema, including `Annotated[list, add]`
on append-only fields (`tool_errors`, `audit_log`). All named constants live in `config.py`.
**Exit check:** `pytest tests/test_state.py` passes; no module imports a literal threshold.

### Phase 2 — Tools layer (degrade paths) ✅
**Build order:** `tool_wrapper` first, then `repo_reader` (repo unreadable = fatal),
`test_parser`, `coverage_parser` (+ static call-graph fallback), `ci_history`,
`test_management`, `vector_store` (Chroma), `sandbox` (subprocess syntax/import check, never
runs against production). Each degrade rule appends to `tool_errors` instead of crashing.
**Exit check:** killing a dependency (e.g. delete the CI fixture) returns a degraded result,
not a traceback.

### Phase 3 — NLP layer ✅
**Files:** `embeddings` (load once, reuse), `similarity`, `clustering`, `extraction`.
Embeddings + cosine + nearest-neighbours; criteria↔test matching and gap detection by
threshold; agglomerative duplicate clustering; spaCy NER + TF-IDF log triage. All
thresholds from `config.py`.
**Exit check:** on sample data, links a test to its criterion, surfaces the no-test
criterion as a gap, and clusters the login duplicate.

### Phase 4 — Analysis spine (linear, no HITL yet) ✅
**Files:** `intake/coverage/redundancy/retrieval/scoring` + scoring prompt.
Each node is `node(state) -> dict`. Intake isolates unparseable tests rather than dropping
them. Scoring calls the reasoning model (strict JSON); missing data → "insufficient
evidence".
**Exit check:** running the spine on `sample_data/` reproduces the golden findings + scorecard.

### Phase 5 — HITL & long-term memory ✅
**Files:** `src/hitl/interrupts.py`, `src/memory/store.py`.
Three interrupt payload builders (removals+evidence → `approved_removals`; ranking →
`approved_priority`; generated tests → `approved_generated_tests`). Memory: `save_decision`/
`get_prior_decisions` (never re-suggest a rejected change), `record_flaky`/
`get_protected_tests`. Interactive blocks; automated auto-approves the recommended set.
**Exit check:** a payload surfaces, blocks, and resumes carrying the decision into state.

### Phase 6 — Decision & generation nodes + the two bounded loops ✅
**Files:** `prioritisation.py` (incl. `coverage_floor_gate` + `revise_node`),
`gap_generation.py`, `validation.py` (incl. `route_after_validation` + `drop_failing_node`)
+ prompts.
- **Coverage-floor gate (Blocker #2):** recompute projected coverage after removals; below
  target → `revise` (pare back the least-valuable removal, re-check). Risk-area tests pinned.
  The gate must *block*, not warn.
- **Bounded validation loop (Blocker #1):** `gap_generation` increments `gen_retry_count`;
  `route_after_validation` → `gap_gen` (retry) / `drop_failing` (after `MAX_GEN_RETRIES`) /
  `approve_tests`. The run always proceeds.
**Exit check (unit, pre-wiring):** a floor-breaching removal set routes to `revise`; perpetual
validation failure lands on `drop_failing` after exactly 3 retries.

### Phase 7 — Assemble & report ✅
**Files:** `assemble.py`, `report.py`. Assemble the keep/merge/quarantine/re-tier plan
side-by-side with the current suite (no external calls). Report renders the four
deliverables, surfaces `tool_errors`, and writes accepted/rejected decisions + new flaky
tests to memory.
**Exit check:** four artifacts render from a hand-built state; every `tool_errors` entry
appears in the output.

### Phase 8 — Graph wiring & entrypoint ✅
**Files:** `src/graph.py`, `main.py`. Register all 10 nodes + `revise` + `drop_failing`; the
linear spine; conditional edges for the gate and the validation loop; the three
`interrupt()` checkpoints; `compile(checkpointer=…)`. `main.py` stays thin.
**Exit check:** `python main.py --suite sample_data/sample_suite` runs the whole graph,
pauses at all three checkpoints, resumes on input, writes the outputs.

### Phase 9 — Tests & evaluation (safety regression) ✅
**Files:** `test_coverage_gate.py`, `test_validation_loop.py`, `test_graph_e2e.py`.
Gate test (breaching set → `revise`, never passes; risk-area never removed); loop test
(always-failing validation stops after `MAX_GEN_RETRIES`, falls back to drop-and-flag); e2e
(full run auto-answering the 3 interrupts, asserting four outputs + audit log + findings
match `expected_findings.json`).
**Exit check:** `pytest` green; golden-set assertions guard analysis quality.

### Phase 10 — Demo frontend (React + FastAPI bridge) ◑ backend done
**Backend (done):** `api.py` compiles the graph with a checkpointer keyed by `thread_id`;
`POST /runs` starts a run and returns the first interrupt payload or outputs;
`POST /runs/{id}/resume` submits a HITL decision and resumes; `GET /runs/{id}` returns
state/`audit_log`; `/health`; CORS for the Vite dev origin. **Recommend-only.**
**Frontend (to build):** Vite + React SPA — input panel (pick/paste suite, set goal,
coverage target, risk areas), run + observe (`audit_log` progress), the **three HITL
approval cards** (removals+evidence → ranking → generated tests), the four deliverables, and
a degraded-results banner for `tool_errors`. **No agent logic in `api.py` or `frontend/`.**
**Exit check:** run backend + frontend, approve all three checkpoints in the browser, see the
four deliverables, confirm findings match `expected_findings.json`.
**To resume:** scaffold `frontend/` (Vite + TS), point it at `http://127.0.0.1:8000`, build
the cards against the documented payloads in [OUTPUTS.md](OUTPUTS.md).

### Phase 11 — Hardening & continuous improvement ⬜ pending
Full audit trail on every node/tool/score/approval; reversible/versioned actions; gradual
rollout (5% / one team); API + webhook entrypoints for `automated` run_mode (CI on new
commit / failed pipeline / new criteria); monthly drift re-scan; feed thumbs up/down into
`memory/` so rejected suggestions never recur and autonomy widens per the spec's ladder.
**Exit check:** an automated (non-interactive) run completes via the webhook/API path, and a
rejected recommendation from a prior run is not re-suggested in the next.

## Dependency map (what blocks what)

```
Phase 0 (learn) ─┐
Phase 1 (state/config) ──> Phase 2 (tools) ──> Phase 3 (nlp) ──> Phase 4 (spine)
                                                                      │
Phase 5 (hitl/memory) ───────────────────────────────────────────────┤
                                                                      ▼
                                              Phase 6 (decisions + loops)
                                                                      ▼
                                              Phase 7 (assemble/report)
                                                                      ▼
                                              Phase 8 (graph + main)
                                                                      ▼
                                              Phase 9 (tests/eval)
                                                                      ▼
                                              Phase 10 (demo frontend)
                                                                      ▼
                                              Phase 11 (hardening)
```

Build the layer below before the layer above; never wire the graph (Phase 8) before its
nodes and loops (Phases 4–7) pass their own checks. The demo frontend (Phase 10) and
hardening (Phase 11) are presentation/ops layers over the same compiled graph — they add no
new agent logic.
